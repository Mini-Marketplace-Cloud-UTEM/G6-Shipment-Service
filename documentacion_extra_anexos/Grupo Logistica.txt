No necesitas ser un experto en camiones ni en cadenas de suministro para entender esto. En el contexto de tu proyecto (un Marketplace en la nube), el "Despacho y Logística" es simplemente un servicio que recibe una dirección, crea una orden de envío y va cambiando el estado de ese paquete a medida que avanza el tiempo (ej: Recibido ➡️ En camino ➡️ Entregado).
Como eres el Grupo 6, tienes un rol crítico en el "camino feliz" de la compra. Vamos a desglosar todo con peras y manzanas para que entiendas el negocio, con quiénes te conectas y qué debes construir paso a paso.
1. El Panorama del Negocio: ¿Qué hace tu servicio?
Tu sistema es el equivalente a "Starken", "Chilexpress" o "Blue Express" dentro del Marketplace. Tu servicio no vende productos ni procesa tarjetas; su única responsabilidad es:
1.	Crear un despacho: Cuando un cliente compra algo de forma exitosa, te avisan para que generes una orden de envío (Shipment).
2.	Hacer seguimiento: Mantener guardado en tu base de datos en qué estado está el paquete (PENDING, IN_TRANSIT, DELIVERED).
3.	Notificar los cambios: Avisar al resto del ecosistema cuando el camión virtual "salió a reparto" o cuando el paquete "ya se entregó en la casa".
2. Tus Aliados: ¿Con qué grupos interactúas?
El documento dice que te integras con G5, G9 y G10. Esto funciona exactamente así:
●	Con el Grupo 5 (Pedidos / Order Management): Ellos son tus principales clientes. Cuando un pedido se paga y se confirma en el Grupo 5, ellos te llamarán (vía API REST o evento) para decirte: "Oye Grupo 6, toma este ID de pedido, va a esta dirección en Santiago y pesa 3 kilos. Despáchalo".
●	Con el Grupo 9 (Notificaciones): A ellos les interesas tú. Cada vez que tú cambies el estado del paquete (por ejemplo, cuando pase a ShipmentDelivered), tú vas a publicar un evento. El Grupo 9 va a capturar ese evento para mandarle un correo o notificación simulada al usuario diciendo: "¡Tu paquete ha llegado!".
●	Con el Grupo 10 (Reportería / Batch / Streaming): Ellos consumen tus datos para armar un Dashboard de ventas y rendimiento global. Querrán saber cuántos despachos se entregaron a tiempo y cuántos fallaron.
3. Tus Conceptos Técnicos Obligatorios
El profesor te asignó conceptos clave que debes demostrar implementados:
●	REST: Debes exponer endpoints tradicionales para que el G5 te pida despachos y el Front (G1) consulte el estado del envío.
●	Eventos: Tu servicio debe ser capaz de emitir mensajes al ecosistema (ej: ShipmentCreated, ShipmentDelivered).
●	Consistencia Eventual: Como el despacho toma tiempo real (un camión no llega instantáneamente), los estados de tu sistema se sincronizarán con los otros grupos de forma asíncrona mediante eventos. No bloqueas al usuario esperando que el camión llegue.
●	Manejo de Errores: ¿Qué pasa si te piden un despacho con una dirección vacía? ¿O un ID de pedido que no existe? Debes responder con códigos HTTP correctos (como 400 Bad Request o 422 Unprocessable Entity).

Revisando las rúbricas exactas que subiste, este es tu plan de acción:
